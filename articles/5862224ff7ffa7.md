## 印象に残ったセッション

### 1.「AI Agent が変える業界特化サービスの業務と顧客体験」
株式会社ウェザーニューズの墨さんによるセッション。

![](https://static.zenn.studio/user-upload/9ab6832a101c-20260627.jpg)

船乗りの命を守りたいというところから創業されたとのことで、すごい観点だと思った。
今回は数あるサービスの中で、海運事業者向けの統合型航海気象プラットフォーム「SeaNavigator」にAI Agentを導入し、どのように開発をしたかというお話。

専門性があり、船乗りの方の命がかかっているからこそ、特に精度が重要だと思います。
そこをAIがこれだけ精度良く回答できるように設計するのは、とても大変だっただろうなと思いました。

セッションでは、AI Agentを実用レベルの精度に持っていくための工夫が、大きく3つの観点で紹介されていました。

![](https://static.zenn.studio/user-upload/b9a372d708d9-20260627.jpg)

#### ① データモデリング（Text2SQL に適したDB設計）

ユーザーの問い合わせに対して、適切なデータソースの選択 → クエリ生成 → JOIN → 分析 → 回答、という一連の流れが必要になります。
そこで、Text2SQL が失敗する要因を洗い出し、対策を打ったとのこと。

- 単一DBにデータを集約し、AI Agent が探索する対象を絞る
- JOIN を単純化するためにスタースキーマ（※スキーマ名うろ覚えですmm）を採用

「AIに探させる範囲をどう絞るか」が精度向上に効くというのは、なるほどと思いました。

#### ② ユーザー特性に合わせた回答生成（Bedrock AgentCore Memory）

普段はオペレータ（人）が船乗りに気象情報を伝えているそうで、その「オペレータらしさ」を Bedrock AgentCore Memory で再現していました。

#### ③ ドメインエキスパートによる AI Agent の改善（Langfuse）

Langfuse を使い、エンジニアでなくても気軽にプロンプトを直せるようにしたとのこと。

#### 成果
成果もすごく、

- 問い合わせの **16%** を AI Agent が回答
- これまで最大 **2週間** かかっていた分析業務を、**最短15分** で完了させたケースもあったそう

数字で出てくると説得力が違いますね。

特に③の、非エンジニアが自分でプロンプトを直せるという仕組みは印象的でした。
弊社でもCSがお客さんと直接やり取りしているので、エンジニアを介さずにCSの方が気軽に修正できるような仕組みづくりができたら強いなと思いました。
そのためにもまずはAWSの知識をもっと強化していきたいです！！

### 2.「そのプロンプト、Skill にしてみませんか？」
Serverworksさんによるセッション。

![](https://static.zenn.studio/user-upload/61ca43d80053-20260627.jpg)

Claude CodeのSkillのお話でした。
普段業務でも使用していますが、使う側・作る側のポイントや所感などを対話形式で聞く感じのセッションで、とても分かりやすく良かったです。

トリガーの精度については以下らしいです。

1. キーワードでなく「ユーザーが実際に打ちそうな言葉」で書く
2. 除外条件を明示して Skill スコープの境界を守る（Anthropic 公式の `/skill-creator` を使えばここはうまくカバーできる）
3. Claude は構造的に Skill を使わない方向にバイアスがある
   → なので「XXXの場合は必ずこのスキルを使うこと」くらい強めに書くのが丁度いい

特にClaude Codeを最近本格的に使い出した私にとって、
Anthropic公式の `/skill-creator` を使えばうまくカバーできるというのは知らなかったので、しっかり勉強してみようと思っています。
