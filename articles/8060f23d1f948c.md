## はじめに

2026年5月11日、TanStackのパッケージが侵害され、`postinstall` フックで `.env` などの機密情報を窃取する worm 型攻撃が発生しました。
自分たちのプロジェクトは直接の被害を受けませんでしたが、依存パッケージのライフサイクルスクリプトを無条件に実行していた現状に危機感を覚え、対策を導入しました。

この記事では以下について書きます。

- pnpm v11 のセキュリティ設定（`strictDepBuilds` / `minimumReleaseAge` / `blockExoticSubdeps`）とデフォルト値
- v9→v11 のバージョンアップ手順
- `minimumReleaseAgeExclude` の安全な運用ルールと CI での自動検知

---
## 攻撃手法の進化

npmサプライチェーン攻撃は世代を追って手口が変わっています。

- `preinstall` / `postinstall` スクリプトで悪意あるコードを実行
- AIコーディングエージェントの `SessionStart` フックやIDEのフォルダ開きタスクを悪用
- [`binding.gyp` を悪用して `node-gyp` 経由でコードを実行](https://snyk.io/jp/blog/node-gyp-supply-chain-compromise-self-propagating-npm-worm-binding-gyp/)。ライフサイクルスクリプト無効化だけでは防げない

またTanStack攻撃では、正規のGitHub Actionsランナーから有効な OIDCトークンを使って悪意あるパッケージが公開されたため、SLSAプロベナンスの検証をパスしました。プロベナンス単独をセキュリティの根拠にすることの限界が示された事例です。

## 自分たちのプロダクトの状況を調査した

対策を検討する前に、自分たちのプロジェクトがどこを突かれうるか確認しました。

**ライフサイクルスクリプトを持つ依存の確認**

`pnpm-lock.yaml` で `requiresBuild: true` なパッケージを探しましたが、該当はありませんでした。
ただし v9 にはそもそもライフサイクルスクリプトを無効化する機能自体がないため、これは「安全」ではなく「把握できていなかっただけ」という状態です。

**git 依存・tarball 直接指定の確認**

`package.json` と `pnpm-lock.yaml` を確認し、直接の git 依存・tarball 直接指定がないことを確認しました。
なお `blockExoticSubdeps` は推移的依存のみが対象なので、直接依存については引き続きレビューで確認する運用が必要です。

**結論**

v9.x 自体にライフサイクルスクリプトの保護機能がなく、依存パッケージが何のスクリプトでも実行できる状態でした。
v11 へのバージョンアップで保護を得るのが最も効果的という判断に至りました。

## pnpm v11のSecurity by Default

[pnpm v11（2026年4月28日リリース）](https://pnpm.io/blog/releases/11.0)では、上記の攻撃ベクトルに直接対応するセキュリティ設定がデフォルトで有効化されました。ライフサイクルスクリプトの原則ブロック、公開直後パッケージの排除、エキゾチックな依存元の遮断、という三段構えです。

| 設定 | 効果 | デフォルト値 |
|------|------|------|
| `strictDepBuilds` | `allowBuilds` 未登録の依存のビルドスクリプトを実行しない | `true` |
| `minimumReleaseAge` | 公開から N 分未満のパッケージバージョンを取り込まない | `1440`（1日） |
| `blockExoticSubdeps` | Git URL / tarball 直接指定の**推移的**依存をブロック（直接依存はカバーしない） | `true` |

私たちは `minimumReleaseAge` を `10080`（7日）に設定しました。
「公開から検出・削除まで数時間〜数日」という攻撃の特性を考えると、7日のクールダウンで多くのworm型攻撃を水際で止められます。

## v9→v11 バージョンアップ手順

1. `package.json` の `packageManager` を更新
   ```
   "packageManager": "pnpm@11.7.0"
   ```

2. Node.js を v22.13 以上に更新
   [pnpm v11 は Node.js 22 以上が必須ですが](https://pnpm.io/blog/releases/11.0)、v11.7.0 では v22.13 以上が要求されます（v22.12.0 では `ERROR: This version of pnpm requires at least Node.js v22.13` が出てインストールできませんでした）。
   バージョンが古い場合はバージョンマネージャー（nodebrew / nvm 等）でアップデートします。
   ```
   nodebrew install v22.17.1 && nodebrew use v22.17.1
   ```

3. corepack で pnpm を有効化
   ```
   corepack enable && corepack prepare pnpm@11.7.0 --activate
   ```

4. codemod を実行
   ```
   npx codemod run pnpm-v10-to-v11
   ```
   v9からでも設定ファイルの機械的変換は有効です。
   ただし今回はTTY必須のため非インタラクティブ環境では動作せず、手動で移行しました。

5. `pnpm-workspace.yaml` を作成
   v11では設定が `package.json#pnpm` / `.npmrc` から `pnpm-workspace.yaml` に集約されます。
   ```yaml
   overrides:
     "@swc/helpers@~0.4": "0.4.36"
     # ...

   publicHoistPattern:
     - "*"
   shamefullyHoist: true  # .npmrc から移行

   strictDepBuilds: true
   minimumReleaseAge: 10080  # 7日
   blockExoticSubdeps: true

   allowBuilds:
     lefthook: true
     "@swc/core": true   # Next.js トランスパイラ（ネイティブバイナリ）
     core-js-pure: true  # polyfill ビルド
     esbuild: true       # バンドラー
     sharp: true         # Next.js 画像最適化（ネイティブバイナリ）
   ```

6. `pnpm install` → `Ignored build scripts` を確認
   `strictDepBuilds: true` により、`allowBuilds` に未登録のパッケージのビルドスクリプトはサイレントにスキップされます（install 自体は正常終了）。スキップされたパッケージは `Ignored build scripts: xxx` としてログに出るので確認できます。
   ローカルではv9時代のキャッシュがあったため問題が出ませんでしたが、CIのfresh環境で初めて顕在化しました。
   `@swc/core` / `sharp` などネイティブバイナリを持つパッケージを `allowBuilds` に追加して解決しました。

## `--frozen-lockfile`でも`minimumReleaseAge`は再検証される

当初「CIでは`--frozen-lockfile`を使っているため `minimumReleaseAge`は効かない」と想定していました。
しかし`pnpm v11.1.3`以降でこの挙動が変わっていました（[#10438](https://github.com/pnpm/pnpm/issues/10438)）。

### 変更前（v11.1.3 より前）

`pnpm install --frozen-lockfile`はロックファイル内の既存エントリに対して`minimumReleaseAge`をスキップしていた。
一度`lockfile`に入ったパッケージはageチェックを素通りしていた。

### 変更後（v11.1.3 以降）

`--frozen-lockfile`でも`tarball`フェッチ前にageを再検証する。

#### 検証方法

`minimumReleaseAge`を190年（99999999）に一時変更して`pnpm install --frozen-lockfile`を実行しました。

```
✗ Lockfile failed supply-chain policy check (2017 entries in 9.6s)
[ERR_PNPM_MINIMUM_RELEASE_AGE_VIOLATION] 2017 lockfile entries failed verification:
  @adobe/css-tools@4.4.1 was published at 2024-11-14T17:48:12.091Z, within the minimumReleaseAge cutoff (1836-04-30T19:26:55.185Z)
  ...and 1997 more
```

`resolution step is skipped`と表示されても、supply-chain policiesの検証ステップは独立して実行されます。
CIでも `minimumReleaseAge` が機能していることが確認できました。

なお`trustLockfile: true`を設定するとこの再検証をスキップできますが、今回は設定せずデフォルトのまま（再検証あり）にしています。

## 緊急パッチ用の`minimumReleaseAgeExclude`運用ルール

例えば、「重大な脆弱性が今さっき修正されたが、7日待てない」というケースのための抜け道です。
ただしこの抜け道自体が攻撃の狙い目になりやすいため、CTOと相談して厳格なルールを設けました。

```yaml
# pnpm-workspace.yaml
# minimumReleaseAgeExclude:
#   - axios@1.7.9 # EXPIRE: 2026-07-01 (CVE-XXXX)
```

### 運用ルール
1. CTOの承認を必須とする（1人の判断で追加しない）
2. バージョン込みで指定する（パッケージ名のみの指定禁止）。[`minimumReleaseAgeExclude` はバージョン指定に対応しており](https://pnpm.io/settings#minimumreleaseageexclude)、名前だけより「修正したいバージョンだけ除外」のほうが穴が小さい
3. チケットに CVE/GHSA 番号など根拠を明記する
4. `# EXPIRE: YYYY-MM-DD`コメントを必ず付ける。v11.1.3 以降は CI の `--frozen-lockfile` でも age が再検証されるため、除外エントリはパッケージ公開日 + 7日（`minimumReleaseAge` を経過する日）まで残す必要がある

### 期限切れをCIで自動検知

削除忘れを人手に依存しないよう、
`.github/scripts/check-expired-excludes.sh`で期限切れを自動検知しました。

```bash
#!/bin/bash
set -euo pipefail

WORKSPACE_FILE="$(dirname "$0")/../../pnpm-workspace.yaml"
if [[ ! -f "$WORKSPACE_FILE" ]]; then
  echo "::error::ファイルが見つかりません: $WORKSPACE_FILE" >&2
  exit 1
fi

# UTC に統一（EXPIRE 日付も UTC 基準で記載すること）
TODAY=$(date -u +%Y-%m-%d)
FOUND_EXPIRED=0

while IFS= read -r line || [[ -n "$line" ]]; do
  if [[ "$line" =~ EXPIRE:[[:space:]]*([0-9]{4}-[0-9]{2}-[0-9]{2}) ]]; then
    EXPIRE_DATE="${BASH_REMATCH[1]}"
    if [[ "$EXPIRE_DATE" < "$TODAY" || "$EXPIRE_DATE" == "$TODAY" ]]; then
      echo "::error::期限切れの minimumReleaseAgeExclude: $line (期限: $EXPIRE_DATE, 今日: $TODAY)" >&2
      FOUND_EXPIRED=1
    fi
  fi
done < "$WORKSPACE_FILE"

if [[ $FOUND_EXPIRED -eq 1 ]]; then
  echo "pnpm-workspace.yaml から期限切れの minimumReleaseAgeExclude エントリを削除してください。" >&2
  exit 1
fi

echo "minimumReleaseAgeExclude 期限チェック OK (チェック日: $TODAY)"
```

`build.yml` の Checkout 直後に仕込んでおくことで、`pnpm install` より前に期限切れを検知してPRを止められます。

## まとめ

| 対策 | 効果 |
|------|------|
| `strictDepBuilds: true` | 未許可パッケージのビルドスクリプトをブロック |
| `minimumReleaseAge: 10080` | 公開7日未満のパッケージを取り込まない |
| `blockExoticSubdeps: true` | Git URL / tarball 直接指定の推移的依存をブロック |
| Socket.dev 導入 | PR 単位でパッケージの静的解析を自動実行 |
| EXPIRE チェックスクリプト | `minimumReleaseAgeExclude` の削除忘れを CI で検知 |

pnpm v11へのバージョンアップは設定ファイルの整理が主な作業で、コアロジックへの影響はありませんでした。
`--frozen-lockfile`での再検証挙動（v11.1.3〜）も実機で確認できて良かったです。
